## 🚨 Token 超限问题彻底修复

### 发现的问题
- Token Watcher 只检查当前会话，不检查所有会话
- Telegram 会话（独立会话）完全在监控之外
- 双模型调试导致对话量激增，10 分钟内 +67k tokens
- 达到 227k tokens → 超限（227k > 203k）

### 根本原因
**Token Watcher 使用 `openclaw status`，只返回当前运行会话的 token 使用量**
- 无法看到其他独立会话（如 Telegram 会话）
- 多会话架构下，每个会话 token 完全独立
- 三重保护在多会话场景下失效

### 解决方案 ✅

- 检查每个会话的 token 使用量
- 多级阈值监控（70%/75%/80%/85%）
- 自动压缩超限会话（保留 20 条消息）
- 防重复压缩（5 分钟冷却）
- 详细日志记录（memory/global-token-watcher.log）

### 经验教训
1. **多会话架构下的监控**: 每个通道会话的 token 是独立的，需要全局监控
2. **API 选择很重要**: `openclaw status` vs `openclaw sessions list --json`
3. **三重保护的盲点**: 依赖单一会话信息在多会话场景下会失效
4. **快速响应**: 对话爆发式增长（10 分钟 +67k）可能无法及时捕获
